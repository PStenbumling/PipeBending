# v0.5.4 — Geometry Engine Design

> Architecture review by senior software architect.
> This is a DESIGN document — not an implementation.
> Source of truth: README.md + CLAUDE.md

---

## Problem statement (concrete, from audit)

Three categories of duplicated logic in the file today:

1. **Bend deduction decision** — the `if (fromax && 90° && size in table) empirical else CLR` block is copy-pasted verbatim in `sbCalculate`, `swCalculate`, `toCalculate`, `mbGetBendDeduction`. It is **absent** in `calculate()` (Bananböj) — that is the critical bug.

2. **Parallel-offset geometry** — `L = D / sin(α)`, `H = D / tan(α)` is computed identically in `sbCalculate`, `swCalculate`, `toCalculate`. Three copies, no shared function.

3. **Mark accumulation** — `mbComputeMarks(segs, bd)` exists as a proper function but is only called by Flerböj. All other tabs accumulate marks inline without going through any shared path.

The engine design addresses exactly these three problems and nothing else.

---

## Data Model

Two primitives. Plain JS objects — no classes, no prototype chains.

```js
/**
 * Segment — a straight run of pipe.
 * @property {number} length  Physical length in mm (what the plumber measures)
 */

/**
 * Bend — the curved section connecting two segments.
 * @property {number} angle_deg   Bend angle (degrees)
 * @property {'fromax'|'manual'}  tool
 * @property {number} size        Pipe diameter mm (needed for empirical lookup)
 * @property {number} clr         CLR mm (ignored when empirical applies)
 *
 * All four fields must be present. The engine decides which model to use.
 * The caller never decides.
 */

/**
 * PipePath — full description of one pipe.
 * Invariant: bends.length === segments.length - 1
 *
 * @property {Segment[]} segments
 * @property {Bend[]}    bends
 */
```

**Constraint:** A PipePath is always a linear sequence — no branches, no loops. This matches every tab in the current app. A PipePath is not a rendering object; it contains only physical data.

## Bend Deduction Strategy

One function. One call site for the decision. No duplication.

```js
/**
 * Compute the bend deduction for a single bend.
 *
 * Returns an object so callers can show which model was used
 * (e.g. the Flerböj badge already does this).
 *
 * @param {Bend} bend
 * @returns {{ value: number, model: 'empirical' | 'geometric' | 'invalid' }}
 */
function geBendDeduction(bend) {
  // Addition (v0.5.4 refinement): guard against missing or malformed Bend objects.
  // angle_deg: 0 is geometrically impossible for a real bend, so !bend.angle_deg
  // correctly rejects both 0 and NaN/undefined. clr == null catches both null and undefined.
  if (!bend || !bend.angle_deg || !bend.size || bend.clr == null) {
    return { value: 0, model: 'invalid' };
  }

  // Addition (v0.5.4 refinement): float-safe 90° comparison.
  // Angles derived from Math.atan() are never exactly 90.0 in floating point.
  // Bananböj already uses this pattern (Math.abs < 0.0001). Use 0.01° tolerance
  // here to be safe across all callers. No practical bend angle is between 89.99°
  // and 90.01° without being intended as 90°.
  const isNominal90 = Math.abs(bend.angle_deg - 90) < 0.01;

  if (
    bend.tool === 'fromax' &&
    isNominal90 &&
    MB_EMPIRICAL_90[bend.size] !== undefined
  ) {
    return { value: MB_EMPIRICAL_90[bend.size], model: 'empirical' };
  }
  const rad = bend.angle_deg * Math.PI / 180;
  return { value: bend.clr * (1 - Math.cos(rad)), model: 'geometric' };
}
```

**Why this prevents model mixing:**
- The decision is made in one place only
- The return object carries the `model` flag — callers can log or display it, but cannot alter the logic
- `'empirical'` and `'geometric'` are mutually exclusive branches; the function never adds or averages them
- Adding a new tool type requires a change in this function only
- `'invalid'` is a third model state — signals bad input without crashing

**Why this fixes the Bananböj bug:**
Bananböj currently computes `clr * (1 - cos(α))` unconditionally at line 3093. Once `calculate()` calls `geBendDeduction()` instead, the empirical path activates automatically for 90°+Fromax. No other change needed.

**Addition (v0.5.4 refinement) — tool mismatch detection:**
`geBendDeduction` itself does not detect tool mismatch — it trusts the Bend object. Detection belongs in Stage 1 (input validation), where the tab knows both `tool` and the raw CLR field value. Each tab that has a Fromax/Manual toggle must perform this check after reading inputs and before building the Bend object:

```js
// Pseudocode — Stage 1, tab-specific
const fromax_clr_for_size = FROMAX_CLR[size];
if (tool === 'manual' && clr === fromax_clr_for_size) {
  showSuggestion('tab-warning',
    'CLR-värdet verkar vara Fromax-förvalt.',
    ['Manuellt läge: ange faktisk CLR för ditt bockarverktyg', 
     `Fromax ${size}mm CLR = ${fromax_clr_for_size} mm — stämmer det?`]
  );
  // Do NOT return — calculation continues with the entered CLR.
}
```

This is a non-blocking suggestion. The geometry engine receives whatever CLR the user has entered; the tab is responsible for flagging if it looks suspicious. No geometry changes.

## Parallel Offset Geometry

```js
/**
 * Geometry for a parallel offset (S-bend) or any single angled section.
 *
 * @param {number} D          Lateral offset mm (sidmått, always CC)
 * @param {number} angle_deg  Bend angle
 * @returns {{ L: number, H: number }}
 *   L = diagonal pipe length between the two bend marks
 *   H = axial projection (along pipe axis) of the diagonal section
 *
 * Addition (v0.5.4 refinement): returns Infinity sentinel values for
 * geometrically degenerate angles. Callers must check before writing to DOM.
 */
function geParallelOffset(D, angle_deg) {
  // Addition (v0.5.4 refinement): guard against division-by-zero.
  // sin(0°) = 0, tan(0°) = 0 → L and H become Infinity.
  // Not reachable through current UI (smallest preset is 15°, autofix minimum is 15°),
  // but required for safety if a future direct-entry path allows arbitrary angles.
  // 180° is also degenerate (sin(180°) ≈ 0 in float).
  if (angle_deg < 1 || angle_deg > 179) {
    return { L: Infinity, H: Infinity };
  }
  const rad = angle_deg * Math.PI / 180;
  return {
    L: D / Math.sin(rad),
    H: D / Math.tan(rad),
  };
}
```

**Caller responsibility:** Any tab using `geParallelOffset` must check `L !== Infinity` before writing results to the DOM. If `L === Infinity`, show a suggestion: "Vinkeln är för extrem — välj mellan 1° och 179°." Do not proceed to Stage 3.

Used by: S-böj, Svanbock (two calls), Takoffset. All three currently compute this inline. No mathematical change for valid inputs — identical result.

## Mark Generation

```js
/**
 * Compute all bend mark positions on an unbent pipe.
 *
 * @param {number[]} segLengths      Physical segment lengths, mm
 * @param {number[]} deductionValues Per-bend deduction values, mm
 *                                   Length must equal segLengths.length - 1
 * @returns {number[]}               Mark positions, one per bend.
 *                                   Last value is total cut length.
 *                                   Returns [] on length mismatch (see below).
 *
 * Algorithm (matches README / CLAUDE.md spec exactly):
 *   mark[0] = seg[0]
 *   mark[i] = mark[i-1] + seg[i] - deduction[i-1]   (i > 0)
 *   total   = mark[last]
 *
 * Edge case: geComputeMarks([300], []) returns [300] correctly.
 * A single segment with no bends is valid — it means "cut to length, no bending."
 */
function geComputeMarks(segLengths, deductionValues) {
  // Addition (v0.5.4 refinement): safe error return instead of throw.
  // A throw in a browser JS event handler is silently swallowed by the event loop —
  // the user sees a blank tab with no explanation. An empty return allows the caller
  // to detect failure and show a suggestion instead.
  if (deductionValues.length !== segLengths.length - 1) {
    console.error(
      `geComputeMarks: expected ${segLengths.length - 1} deductions, got ${deductionValues.length}`
    );
    return [];
  }
  const marks = [];
  let pos = 0;
  for (let i = 0; i < segLengths.length; i++) {
    pos += i === 0 ? segLengths[i] : segLengths[i] - deductionValues[i - 1];
    marks.push(pos);
  }
  return marks;
}
```

**Caller responsibility:** Check `marks.length > 0` before reading `marks[i]`. If empty, show: "Internt fel — kontrollera att alla sträckor är ifyllda." Do not write to DOM.

Mathematically identical to the current `mbComputeMarks(segs, bd)` when all deductions are the same scalar. Generalizes to per-bend deductions without changing the algorithm. Last element of the returned array is total cut length.

## Clamp Length Safety

> Addition (v0.5.4 refinement)

Fromax No.55 and similar ratchet benders require approximately `2 × OD` of straight pipe between a bend mark and the pipe end in order to grip without the pipe slipping or being distorted.

| Pipe size | Min clamp length |
|---|---|
| 12 mm | 24 mm |
| 15 mm | 30 mm |

M1 = 0 is currently accepted everywhere. It is mathematically valid and produces correct mark calculations. It is physically impossible — the tool has nothing to hold. This is the most likely source of "almost correct" field results.

```js
/**
 * Check whether a pipe segment is long enough for the bender to grip safely.
 *
 * @param {number} segLength  Length of the segment in mm
 * @param {number} pipeSize   Pipe OD in mm (e.g. 12 or 15)
 * @returns {{ ok: boolean, minRequired: number }}
 */
function geCheckClampLength(segLength, pipeSize) {
  const min = 2 * pipeSize;   // 2 × OD — minimum grippable length
  return { ok: segLength >= min, minRequired: min };
}
```

**Usage — Stage 4 (tab-specific, non-blocking):**
After marks are computed, call `geCheckClampLength` for:
- The first segment (length before M1): the lead-in that sits in the tool jaws
- The last segment (length after final mark): the tail the plumber holds while removing the tool

```js
// Pseudocode — Stage 4, per tab
const leadIn  = geCheckClampLength(M1, pipeSize);
const tailOut = geCheckClampLength(rB, pipeSize);

if (!leadIn.ok) {
  showSuggestion('tab-warning',
    `Röret kan vara för kort framför M1 för att bockarverktyget ska få grepp.`,
    [`Minst ${leadIn.minRequired} mm rekommenderas — nu ${M1.toFixed(0)} mm`,
     'Lägg till mer rör före M1, eller byt till ett verktyg med kortare klämkäkar.']
  );
}
if (!tailOut.ok) {
  showSuggestion('tab-warning',
    `Röret är väldigt kort efter sista böjen.`,
    [`Minst ${tailOut.minRequired} mm rekommenderas — nu ${rB.toFixed(0)} mm`,
     'Det kan vara svårt att hålla röret stabilt vid sista böjen.']
  );
}
```

**Rules:**
- These are suggestions, not blockers. Calculation proceeds regardless.
- `showSuggestion` follows the existing UX pattern — same CSS class, same suggestion engine.
- Do not show both warnings stacked unless both are truly violated. Use the lead-in check first; if it fails, skip the tail check to avoid message overload.
- `geCheckClampLength` has no side effects. It only returns data.

## Calculation Flow (per tab)

Every tab follows the same four-stage pipeline. The engine handles Stages 2 and 3. Safety checks belong in Stage 1 (input) and Stage 4 (output) — they are tab-specific and never inside the engine.

```
Stage 1 — Input collection (tab-specific, DOM reads)
  Read raw values from inputs
  Validate required fields (D/sidmått is the only truly required field for offset tabs)
  [Addition v0.5.4] Check for tool mismatch (manual CLR = Fromax preset) → suggestion
  Show suggestion if invalid → return early

Stage 2 — Geometry (engine functions, pure math, no DOM)
  Build the Bend object(s) with all four fields
  Call geBendDeduction(bend) for each bend
    → check model !== 'invalid' before continuing
  Call geParallelOffset(D, angle) if applicable
    → check L !== Infinity before continuing

Stage 3 — Mark generation (geComputeMarks, pure)
  Build segLengths array from Stage 1 + Stage 2 results
  Build deductionValues array from Stage 2 results
  marks = geComputeMarks(segLengths, deductionValues)
    → check marks.length > 0 before continuing

Stage 4 — Output (tab-specific, DOM writes)
  Write marks[i] to result elements
  Draw SVG diagram
  [Addition v0.5.4] Call geCheckClampLength for first and last segments → suggestion
  Show success/warning
```

The engine owns Stages 2 and 3. Stages 1 and 4 remain tab-specific and are not touched during migration.

## Do NOT do

- **Do not add a class or object prototype.** The file has no module system. Plain functions are sufficient.
- **Do not create a unified `Pipe` model that all tabs share.** Each tab's input geometry is different enough that a forced shared model adds abstraction without removing code.
- **Do not change any input field IDs, names, or DOM structure.** The engine is a JS-only concern.
- **Do not add unit test infrastructure in this step.** Manual test cases are sufficient. A test runner is a separate, later concern.
- **Do not migrate all tabs in one commit.** Each step must be independently verifiable and independently revertable.
- **Do not expose `model: 'empirical'|'geometric'` in the UI.** Available for internal use only (Flerböj badge already handles this via `mbIsEmpirical()`).
- **Do not modify `geComputeMarks` to handle non-linear pipe paths.** That is a future problem.
- **Do not rename `MB_EMPIRICAL_90` or `FROMAX_CLR`.** They are referenced throughout the file and in CLAUDE.md.
- **Do not block calculation on clamp length or tool mismatch warnings.** These are suggestions only. The plumber decides.
- **Do not add springback compensation to S-böj, Svanbock, or Takoffset in this version.** That is a v0.5.5 item.
