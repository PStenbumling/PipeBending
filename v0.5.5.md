## Purpose

v0.5.4 ensured the app calculates correctly.
v0.5.5 ensures the plumber uses those calculations correctly.

**The problem this version solves:**
A correct answer presented to the wrong person at the wrong moment is still a wrong pipe.

Validated failure modes from the field — all of which v0.5.4 does not address:

- A plumber measures M1 from the wrong end. Pipe is cut wrong by 2×M1.
- A plumber uses a no-name bender with CLR=38mm but the app assumes 43mm. Offset is 5mm off. He refits. Fittings stress-crack after two weeks.
- A plumber clears the lead-in field to zero. Pipe is cut correctly. The bender has nothing to grip. The pipe slips at 30° and is scrapped.
- An apprentice reads "sidmått 60mm" and measures 60mm along the pipe diagonal instead of perpendicular. The S-bend ends up short.
- A plumber taps through the form quickly on a phone on a scaffold. He misses a yellow warning. He bends. He discovers the problem at the fitting.

None of these are math errors. All of them produce wrong pipe.

**Why it is needed after v0.5.4:**
v0.5.4 introduced a safety layer at the geometry engine level — clamp length checks, tool mismatch detection, float-safe comparisons. These fire at calculation time. v0.5.5 moves the protection earlier, makes it visible under stress, and ensures it persists until dismissed by action — not by scrolling past it.

## Design Principles

### 1. Prevent before calculate, not after

A warning shown after a result is processed is a warning that will be ignored. Validation that happens on input change — before the user has mentally moved on — is acted on. The earlier the signal, the cheaper the correction.

### 2. Do not rely on user attention

A plumber on a scaffold in winter with gloves on and a phone screen at 40% brightness will not read a yellow box in the middle of a result panel. Design for the worst case: assume the user is stressed, moving fast, and will not read anything that is not in their direct visual path.

### 3. Warnings must state consequences, not conditions

"H är för litet" tells the plumber nothing actionable.
"Bocken får inte plats — röret slinter i verktyget" tells him exactly what will go wrong and why to stop.

Every warning must answer: **what will happen if I ignore this?**

### 4. Never block unnecessarily

A blocked UI on a scaffold is a hostile UI. The plumber will find a workaround — usually ignoring the app entirely. Every safety check is a suggestion with a stated risk, never a hard stop. The app trusts the plumber. The app informs; the plumber decides.

### 5. Warnings must persist until the condition is resolved

A warning that disappears when the user scrolls or taps elsewhere has zero effect on stressed users. Warnings tied to specific input fields must reappear every time that field is interacted with, until the value is within the safe range.

### 6. Visual reinforcement over text

For users who scan rather than read: color, icon, and position communicate faster than sentence structure. The warning color, border, and placement must be consistent enough that the plumber learns to recognize "red = something is wrong with this measurement" without reading the text.

### 7. Estimated values must be visibly marked as estimates

A plumber who trusts a value that is marked as estimated and later discovers it was wrong loses trust in the entire tool. Estimated values (currently: 15mm empirical deduction = 27mm) must carry a persistent visual marker that survives into the results — not just into the input area.

### 8. Physical impossibility overrides UX softness

If something cannot physically work, the UI must say so explicitly.

Not:
"rekommenderas"

But:
"detta fungerar inte"

Reason:
A plumber accepts risk — but not impossibility.

### 9. Rounding must be internally consistent

All visible values must be derived from the same rounded reference.

- Kaplängd must ALWAYS equal the last mark position (rounded)
- Users must never need to calculate distances between marks
- Marks are absolute positions, not segment lengths

Reason:
Mixing rounded cumulative marks with independently rounded lengths creates invisible 1–2 mm drift in real pipe.

## Safety Layers

Each layer addresses one validated failure mode. Layers are independent — adding or removing one does not affect the others.

Refinement (v0.5.5 review):

Clamp length violations must be treated as physically impossible.

- Use high-contrast warning (not standard suggestion)
- Must include consequence:
  "Röret kommer glida i verktyget – måttet blir fel"

- Results may still be shown, but must appear visually unsafe

Reason:
Users ignore normal warnings, but react to physical impossibility.

### Layer 1 — Reference direction

> **Refinement (v0.5.5 review) — reference dominance (critical):**
>
> Reference direction must dominate the entire UI.
>
> - The reference end (START / H-stycke) must be visible at ALL times
> - It must be impossible to scroll past it
> - It must appear BOTH in input zone and result zone
>
> If the reference is not visible, the system is unsafe.
>
> Reason:
> A wrong reference produces the largest possible error (2×M1) and is not detectable until installation.
>
> Implementation rule:
> The reference strip is not tied to results.
> It must be rendered:
> - On tab load
> - Above inputs (sticky)
> - Above results (duplicate or same component)
>
> The user must always see:
> → where 0 mm is

**Consequence message (if mark positions look suspicious — M1 > total/2):**
> "M1 är mer än halva rörlängden — är referensänden rätt? Kontrollera att du mäter från [ände]."

Refinement — reference anchoring in Bananböj:

- The reference end must be explicitly labeled in results:
  "Kapa här (0 mm – H-stycke)"

- The opposite end must NOT visually compete as origin
- Instick visuals must not imply measurement origin

Reason:
Users default to visually dominant elements (fittings) over abstract reference strips.

### Layer 2 — Tool mismatch

Refinement (v0.5.5 review):

Tool mismatch warnings must be persistent.

- Must remain visible after calculation
- Must stay until user corrects tool or CLR
- Must include consequence:
  "Röret hamnar fel — installationen kommer inte passa"

Reason:
Transient warnings are ignored under stress.

**Risk:** Plumber switches to manual tool mode but does not update CLR. App uses Fromax-preset CLR (43mm for 12mm pipe). Actual tool CLR differs. Bend deduction is wrong. Offset is physically off. Error is silent — no indicator, no warning.

**When it triggers:** When `tool === 'manual'` AND the CLR field value matches `FROMAX_CLR[selectedSize]`.

**What the system does:**

1. The CLR input field receives a yellow border and a persistent inline note directly below it:
   > "CLR ser ut som Fromax — om ditt verktyg har annan radie hamnar röret fel"

   This note is not in a warning box — it is attached directly to the CLR field. It persists as long as the condition holds. It disappears when the user changes CLR to a value that does not match any entry in `FROMAX_CLR`.

2. In the result panel, if the mismatch condition holds at calculation time, a badge appears next to the bend deduction result.

   > **Refinement (v0.5.5 review) — concrete consequence:**
   > The badge text must not describe the condition — it must describe what happens in the real world if the condition is ignored. Replace abstract "kontrollera om det stämmer" with:
   > "⚠ Fel böjradie → röret hamnar på fel ställe. Offseten blir fel och kopplingarna spänns."
   >
   > Rationale: "kontrollera" is an instruction the stressed user skips. "röret hamnar på fel ställe" is a consequence that lands. The difference is: one asks the user to do work, the other tells the user what they lose if they don't.

3. No calculation is blocked. The plumber may knowingly use Fromax CLR with a manual tool.

**Consequence message (result badge):**
> "⚠ Fel böjradie — offseten blir fel och kopplingarna kan spännas. Ange faktisk CLR för ditt verktyg."

### Manual override reasonableness guard

If manual bend deduction is provided:

- If value > pipeSize × 3 → show warning:

"Ovanligt stort böjavdrag — kontrollera värdet"

- If value matches CLR instead of deduction → suggest correction

Reason:
Users commonly confuse CLR and deduction, causing large systematic errors.

### Layer 3 — Clamp length

**Risk:** Lead-in (pipe before M1) or tail (pipe after last mark) is shorter than the bender's minimum grip length (≈ 2×OD). The pipe slips in the tool. The bend point shifts. The pipe is scrapped or must be recut.

**Threshold (from `geCheckClampLength` in v0.5.4):**
- 12mm pipe: min 24mm
- 15mm pipe: min 30mm

**When it triggers:** On every calculation, in Stage 4, for the first segment (M1 value) and the last segment (tail value). Also on `oninput` for the M1 and tail fields if they are below threshold.

**What the system does:**

1. The M1 input field shows a red border and inline warning when `M1 < 2×pipeSize`:
   > "Kort grepp — bocken slinter. Minst Xmm krävs för att fungera."

   This appears on input change, not only on calculate. It is the earliest possible signal.

2. In the result panel, if the condition holds, a red inline note appears immediately above the marks table:
   > "⚠ Fysiskt omöjligt — röret slinter i verktyget. Bara Xmm framför M1 (minst Ymm krävs)."

3. For the tail segment: same pattern, shown below the last mark row.

4. The warning persists in the result panel on every recalculation as long as the condition holds.

> **Refinement (v0.5.5 review) — clamp prioritization:**
> If both lead-in (M1) and tail (rB) are below threshold, show ONLY the lead-in warning.
>
> Rationale: the lead-in failure is more dangerous (the tool slips at the start of the bend, producing a completely wrong pipe) and more common (plumbers often minimize lead-in to save material). Showing two clamp warnings simultaneously splits attention and reduces the urgency of both. One focused warning is acted on. Two competing warnings are ignored.
>
> Implementation rule: in Stage 4, evaluate lead-in first. If `!clampIn.ok`, show lead-in warning and skip tail check entirely. If `clampIn.ok`, evaluate tail. If `!clampOut.ok`, show tail warning. Never show both simultaneously.
>
> The tail warning is a secondary concern — the pipe can usually be removed without distortion even with a short tail; a short lead-in cannot be recovered from.

**Consequence message (lead-in, priority):**
> "Fysiskt omöjligt — röret slinter. Bara Xmm framför M1 (minst Ymm krävs). Flytta M1 längre in eller lägg till mer rör."

**Consequence message (tail, only if lead-in is ok):**
> "Kort svans efter sista böjen — kan vara svårt att hålla stilla. Minst Xmm rekommenderas."

**Non-blocking:** Calculation proceeds. The plumber may have a tool with shorter jaws, or may be using a different clamping method.

### Layer 4 — Offset misunderstanding (sidmått)

**Risk:** Plumber reads "sidmått" and measures the distance along the diagonal pipe rather than the perpendicular cc distance. The result is a sidmått that is larger than reality. The pipe offset is wrong. This is the most common apprentice error.

**When it triggers:** On focus of the sidmått input field (D / sb-offset / sw-offset / to-D). Passive reinforcement only — no active warning unless value looks implausible.

**What the system does:**

1. The sidmått input field carries a permanent illustrated hint below it. Not a text hint — a small inline SVG icon showing two parallel vertical lines with a horizontal double-headed arrow between them, labeled "cc". This requires no text reading. The plumber either recognizes it or asks.

2. The hint text reads: "Mät rakt tvärs — inte längs röret. Centerline till centerline (cc)."

3. **Refinement (v0.5.5 review) — context-aware implausibility check:**

   The original check (`D > total_pipe_length * 0.8`) is too coarse and angle-agnostic. A large D is normal at shallow angles and suspicious at steep angles.

   Replace with an angle-aware check: if `D > H * 1.5` where H is the computed axial projection (`H = D / tan(α)`), the offset-to-run ratio is geometrically extreme. This condition is equivalent to `α > ~56°` for any D — meaning the bend is steeper than 56°, and the sidmått is implausibly large relative to the axial space available.

   Additionally: if `D` was just entered AND the computed `L` (diagonal pipe length) is shorter than `D` — meaning the user entered a sidmått larger than the pipe section it is supposed to traverse — show the check. This catches the "measured along the pipe" error directly: the user measured L and entered it as D.

   Trigger condition: `D >= L * 0.95` (sidmått is nearly as large as or larger than the diagonal pipe section). This is geometrically impossible for angles above ~5° unless the user measured the wrong dimension.

   Message:
   > "Sidmåttet verkar lika stort som rörsträckan — mätte du tvärs eller längs röret? Sidmåttet ska vara avståndet rakt åt sidan, inte längs böjen."

4. For Takoffset specifically: D is computed from `h2 - h1`, so the user cannot enter it directly. No offset confusion is possible in that tab. Skip this layer entirely for Takoffset.

**Consequence message:**
> "Fel sidmått → röret hamnar på fel ställe. Mät rakt tvärs mellan centerlines."

Refinement — shallow angle guard:

If angle α < 20°:
Show inline confirmation:

"Lågvinkel-böj — är sidmåttet verkligen X mm rakt tvärs?"

Reason:
At shallow angles, diagonal vs perpendicular confusion is not caught by geometric plausibility checks.

### Layer 5 — Estimated value visibility

**Risk:** The 15mm empirical bend deduction (27mm) is marked in code as "uppskattad, ej uppmätt." A plumber using 15mm pipe with Fromax trusts this value. If the actual deduction is different, all marks on every pipe in the job are systematically off by the same amount. This is the most dangerous kind of error — consistent, invisible, and cumulative.

**When it triggers:** Whenever `size === 15` AND `tool === 'fromax'` AND bend is 90°. Persistent until the user overrides the bend deduction manually.

**What the system does:**

1. The bend deduction badge (Flerböj already has this) must show for all tabs, not just Flerböj. When the estimated value is active:
   > "⚠ 15mm Fromax 90°: böjavdrag = 27mm — UPPSKATTNING, ej uppmätt. Verifiera med ett provrör."

2. The badge color is amber (warning), not red (error). The value is usable. It is not wrong. It is unverified.

3. A one-time dismissible note appears on first use per session:
   > "Du bockar 15mm med Fromax. Böjavdraget (27mm) är ett uppskattat värde — bocka ett provrör och mät faktiskt avdrag innan du kapar till hela jobbet."

4. If the user overrides the bend deduction manually (manual input field), the badge disappears and is replaced by "Manuellt böjavdrag: Xmm".

> **Refinement (v0.5.5 review) — estimated value fatigue prevention:**
>
> A static amber badge shown on every pipe becomes invisible through repetition. After the first occurrence, the plumber has mentally filed it as "normal" and stops reading it. This is estimated value fatigue — the warning is present but no longer perceived.
>
> Two countermeasures:
>
> **a) Escalating visibility on repeated use within session.**
> Track how many times the estimated deduction has been used in the current session (count in a module-level variable, e.g. `let estimatedUseCount = 0`). On first use: amber badge (normal). On third use and beyond within the same session: the badge text gains a count annotation:
> > "⚠ UPPSKATTNING (använt 3 ggr denna session) — har du kontrollerat med provrör?"
>
> The count annotation is a pattern interruption — it breaks the visual routine the eye has learned to skip.
>
> **b) Audit trail in results.**
> When printing or copying results (future feature), the estimated badge is included verbatim in the output. This is a deferred note — it does not require implementation in v0.5.5, but the badge HTML must be structured so it is not stripped by a `textContent` read. Use a data attribute: `data-estimated="true"` on the badge element.

Refinement — interaction with warning hierarchy:

If a RED (danger) warning is active:

- Estimated value badges must be suppressed in the result view
- The information is not removed, only visually deferred

Reason:
Estimated value warnings must not compete with physically impossible conditions.

## Behavior Design

### Designing for the stressed user

The five behaviors of a stressed user in the field:

1. **Scans, does not read.** Implication: warnings must be visually distinct without reading. Color + icon + position, not paragraph text.

2. **Taps through quickly.** Implication: warnings that require a deliberate tap to dismiss are more likely to be seen than warnings that auto-dismiss. But mandatory dismissal for non-critical warnings is hostile. Use persistence without blocking.

3. **Ignores yellow.** Implication: yellow is overused in web UIs and carries low urgency signal. Use amber for genuine concern, red only for physically dangerous conditions (clamp length, reference direction). Severity must be legible at a glance.

4. **Trusts the most recent number.** Implication: a warning that appears before the result is calculated may be overwritten in the user's mental model when they see the result. Warnings that survive into the result panel — attached to the output numbers, not above the inputs — are more effective.

5. **Will not scroll back up.** Implication: warnings must appear in the output zone, not only in the input zone. A warning above the inputs is invisible once the user has scrolled to the results.

### Warning placement strategy

| Condition | Input zone | Result zone |
|---|---|---|
| Clamp too short (lead-in) | Red border + inline (on input change) | Red note above marks table |
| Clamp too short (tail) | Red border + inline (on input change) | Red note below last mark — **only if lead-in is ok** |
| Tool mismatch | Yellow border + field note | Badge next to deduction value (concrete consequence text) |
| Estimated value (15mm) | Amber badge in CLR area | Amber badge in deduction row, escalates after 3× use |
| Reference direction | Persistent strip (always visible) | Arrow in marks table header |
| Offset confusion | Permanent cc icon below field | — (angle-aware check on input, not in result zone) |

### Warning hierarchy

If any RED (danger) condition exists:

- Suppress all amber advisory badges
- Replace with a single high-priority header:

"⚠ ÅTGÄRDA FEL INNAN MONTERING"

- Red warnings must be visually dominant
- No competing messages allowed

Reason:
Multiple warnings reduce perceived urgency. One clear danger signal is acted upon.

### Timing strategy

| Condition | Trigger timing |
|---|---|
| Clamp too short | On `oninput` (immediate) |
| Tool mismatch | On `oninput` for CLR, on tool toggle |
| Reference direction | On tab entry + on M1 input change |
| Estimated value | On pipe size change + on calculation |
| Offset confusion | On field focus (passive cc icon, always) + angle-aware check on input |

### Persistence rules

- **Red warnings** (clamp, reference): persist until the value is corrected. Reappear on every recalculation if condition holds.
- **Amber warnings** (tool mismatch, estimated): persist until user takes explicit corrective action (changes CLR value, overrides deduction). Do not auto-dismiss.
- **Blue/neutral hints** (offset cc icon, reference strip): always visible. Never dismissible. Part of the field label area, not the warning system.

> **Refinement (v0.5.5 review) — clamp warning priority enforcement in timing:**
> The `oninput` handler for M1 and rB fields must check lead-in first. If M1 triggers a clamp warning, the rB field's warning element is explicitly hidden (`classList.remove('visible')`) even if rB is also below threshold. This must be enforced both in the `oninput` handler AND in Stage 4 of the calculate function — two separate enforcement points — because the user may change M1 after having already scrolled to results.

## UX Patterns

These patterns are reusable across all tabs. They do not introduce new CSS classes — they use or extend the existing `sb-warning` / `sb-success` / suggestion engine from v0.5.3.

### Pattern 1 — Field-level inline note

Attached directly below a specific input field. Appears on input change. Uses the existing `.hint` class styling but with color variant.

```
[ input field                    ]
  ⚠ Fysiskt omöjligt — röret slinter. Minst 24mm krävs
```

Variants:
- `.hint-warn` — amber, for advisories
- `.hint-danger` — red, for physically dangerous conditions
- `.hint` — neutral, permanent (existing, for cc icon etc.)

### Pattern 2 — Sticky result badge

A small tag attached to a specific result value in the marks table or result hero. Not a box — a tag. Appears when the value has a known quality issue.

```
Böjavdrag    27 mm  [⚠ UPPSKATTNING]
```

The badge is part of the result row. It cannot be ignored by scrolling past the warning box.

> **Refinement (v0.5.5 review) — badge escalation for estimated values:**
> Badges for estimated values must support an escalated state after 3 uses within the session. The escalated badge adds a count annotation to interrupt visual routine:
> ```
> Böjavdrag    27 mm  [⚠ UPPSKATTNING — 3× denna session]
> ```
> Implementation: the badge element has a `data-use-count` attribute updated by JS. CSS targets `[data-use-count="3"]` and above with a slightly stronger border or background to differentiate from the baseline badge. One CSS rule addition, no new class.

### Pattern 3 — Persistent reference strip

A narrow, always-visible bar above the marks table. Shows the reference direction as a visual, not a text description. Contains a minimal pipe diagram with an arrow at the measurement origin.

```
← Mät härifrån        M1        M2        [rörets slut]
   (0 mm)          120 mm     205 mm        300 mm
```

This strip replaces the current standalone "Böj vid M1 och M2..." note with something that is spatially anchored to the measurement direction.

### Pattern 4 — Consequence-first messaging

All warning text follows the format:
> "[What will happen] — [why / what the app detected]"

Not:
> "Värdet är för litet"

But:
> "Bocken kan slinta — röret är bara 15mm framför M1 (minst 24mm behövs)"

The consequence leads. The condition follows. This is how a more experienced plumber would warn an apprentice.

> **Refinement (v0.5.5 review) — tool mismatch message upgrade:**
> The current tool mismatch message ("kontrollera om det stämmer för ditt verktyg") is instruction-first, not consequence-first. Replace with consequence-first in all locations:
>
> Field note (input zone): "CLR ser ut som Fromax-förvalet — om ditt verktyg har annan böjradie hamnar röret fel."
>
> Result badge (output zone): "⚠ Fel böjradie → offseten blir fel och kopplingarna spänns."
>
> The result badge must be stronger than the field note — the user who ignored the field note gets a second, more direct consequence statement when they see the results.

### Pattern 5 — One-time session note

Used for high-signal, low-frequency reminders (estimated value, reference direction on first use). Appears once per session per tab. Has a clearly visible dismiss button. Does NOT persist after dismiss within the session. Reappears on page reload.

Implementation: `sessionStorage.getItem('v555_dismissed_X')` — if not set, show. On dismiss, set it. This requires no server, no persistence beyond the session.

## Non-goals

- **No blocking flows.** No calculation is ever prevented. The plumber always gets their numbers. Warnings are information, not gates.
- **No new math.** v0.5.5 does not add, modify, or depend on any geometry calculation. It reads results from the existing engine and decorates them.
- **No new tabs or workflows.** Every safety layer is added to existing tabs using existing UX patterns.
- **No springback.** Springback compensation for additional tabs is a feature, not an error prevention measure. It belongs in v0.5.6 or later.
- **No new tools or bender types.** The tool model (fromax / manual) is fixed in v0.5.4. v0.5.5 does not extend it.
- **No modal dialogs.** Modals block workflow and are especially hostile on mobile. Every warning is inline.
- **No persistent storage of warning dismissals.** Beyond the session-level note pattern (Pattern 5), no warning state is stored in localStorage or cookies. Every session starts clean.
- **No A/B testing or analytics hooks.** Out of scope.

## Implementation Strategy

### How this is added on top of v0.5.4

v0.5.5 is purely additive. It introduces:
1. New CSS classes (`.hint-warn`, `.hint-danger`, sticky badge styles)
2. New HTML elements in existing tab panels (reference strip, field-level notes)
3. New JS calls in existing Stage 1 and Stage 4 positions
4. `sessionStorage` reads/writes for one-time session notes
5. One module-level counter: `let estimatedUseCount = 0` for estimated value escalation

No existing function is modified. No engine function is touched. No input field ID or DOM structure changes.

The `geCheckClampLength` function introduced in v0.5.4 is called in Stage 4 — this is the one v0.5.4 dependency. All other safety layers are independent of the engine.

### Rollout order (per tab)

Roll out one tab at a time. Each tab is independently testable.

**Step 1 — S-böj (highest traffic, simplest structure)**
- Clamp length warning on M1 (priority) and tail (secondary, only if M1 ok)
- Tool mismatch badge on CLR field + consequence-first result badge
- Reference strip above marks table
- cc icon on sidmått field + angle-aware implausibility check

**Step 2 — Svanbock**
- Same as S-böj
- Reference strip is identical in structure

**Step 3 — Takoffset**
- Clamp length on M1 and tail (priority rule applies)
- Tool mismatch badge
- Reference strip (D is computed, no offset confusion risk — skip Layer 4)

**Step 4 — Flerböj**
- Clamp length on first segment and last segment (priority rule applies)
- Estimated value badge for 15mm with escalation counter (already has deduction badge — extend it)
- Reference strip (START-based, consistent)

**Step 5 — Bananböj**
- Reference direction note (H-stycke, different from all other tabs — highest risk)
- One-time session note on first use: "Alla mått räknas från H-styckets ände"
- Estimated value badge with escalation if 15mm is selected

**Step 6 — Takrör**
- Clamp length on s1 and s4 (priority rule: s1 first)
- Estimated value badge with escalation if 15mm is selected

## Feature Track 2 — Visual Instick in Couplings (Bananböj)

> Merged from separate design note v5.5.md.
> This is a UX/visual release. Geometry formulas and mark math stay unchanged.

### Why

Current Bananböj diagram shows bend geometry and mark positions but does not show how much pipe goes into fittings. In practice:
- Apprentices can read the marks but still hesitate at assembly
- The diagram explains where to bend, not how the pipe seats in fittings
- Users may over-trust cut length without validating insertion depth visually

### Goal

Make instick visually explicit so a plumber can answer at a glance:
- How far does the pipe go into T-kopplingen?
- How far does the pipe go into H-stycket?
- Where does free straight pipe start/end relative to the fitting mouth?

### Scope

**In scope:**
- Bananböj diagram: show instick zones for both ends
- Labels in plain Swedish in diagram and result text
- Optional subtle enhancement for S-böj/Svanbock/Takoffset: show "fri rak längd" where applicable

**Out of scope:**
- Any change to deduction model or decision logic
- Any change to core geometry helpers (`geBendDeduction`, `geParallelOffset`, `geComputeMarks`)
- Any change to stock/coupling split math in Flerböj

### Visual design

**1 — Coupling insertion zones**
For each fitting end in the diagram:
- Draw a short highlighted segment inside fitting mouth = instick zone
- Use a distinct color tint (not same as main pipe stroke)
- Add compact label near zone: "Instick T: XX mm" / "Instick H: YY mm"

**2 — Fitting mouth marker**
At each fitting connection point:
- Draw a small marker line where free pipe transitions to inserted pipe
- This separates: inserted pipe (inside fitting) vs free pipe (outside fitting)

**3 — Dimension callouts**
Short dimension arrows/lines for instA (T-koppling) and instB (H-stycke).
Rules:
- Whole mm in visible labels
- Internal precision unchanged
- Labels use plumber language, never variable names (not "instA" — use "Instick T-koppling")

**4 — Legend block (small, optional on narrow screens)**
- Solid pipe = visible/free section
- Highlighted section = instick i koppling
- Marker line = kopplingens mun

### Implementation notes

Data already available in Bananböj: `instA`, `instB`, `M1`, `freeBefore`. No new user inputs.

SVG rendering order:
1. Draw fittings (existing style)
2. Draw inserted segments (instick zones) — rendered first, below free pipe
3. Draw free pipe path
4. Draw mouth markers on top
5. Draw text labels and dimension callouts

Responsive: narrow viewport → hide legend first, keep instick labels and markers always visible.

### Acceptance criteria

1. Diagram clearly shows insertion depth in both ends for Bananböj
2. User can distinguish inserted vs free pipe without reading formulas
3. Visible values are whole mm
4. No changes to calculation outputs
5. Existing validations and suggestion engine behave as before
